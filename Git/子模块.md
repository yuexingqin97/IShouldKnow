# Git Submodule —— 子仓库是怎么工作的

> 我脑子里的模型：submodule 不是把别人的代码"复制"进我的仓库，而是**在我的仓库里留了一个便签**：「这个目录，去这个 URL，拿这个 commit」。

## 我的仓库里长什么样

以 `isk` 为例，`_raw/` 下有三个开源书：

```
isk/                          ← 我的仓库
├── .gitmodules               ← 登记表（记录 URL 和路径）
├── .git/
│   └── modules/              ← 子模块的 git 数据存这里
│       ├── _raw/rust-by-example-cn/
│       ├── _raw/rust-course/
│       └── _raw/typescript-tutorial/
├── _raw/
│   ├── rust-by-example-cn/   ← 看起来像普通目录，其实是个"指针"
│   ├── rust-course/          ← git 不跟踪它的内容，只记一个 commit hash
│   └── typescript-tutorial/
└── Readme.md
```

## 核心概念：submodule 的三块拼图

### 拼图 1：`.gitmodules` —— 登记表

这个文件**明文存在我的仓库里**，记录了每个子模块的路径和 URL：

```ini
[submodule "_raw/rust-course"]
    path = _raw/rust-course
    url = https://github.com/sunface/rust-course.git
```

> `.gitmodules` 跟着我的仓库一起 commit、一起 push。别人 clone 后 git 靠这个文件知道"要去哪下载子模块"。

### 拼图 2：gitlink —— 指针

在 git 的 index（暂存区）里，`_raw/rust-course` 不是一个目录，而是一个 **160000 模式的 gitlink**——它只存了一个 commit hash：

```
# git ls-tree HEAD _raw/rust-course
160000 commit 386a11f17a7118eaea4fd68671520d757877ebfb    _raw/rust-course
```

**关键**：git 不关心 `_raw/rust-course/` 里面有什么文件。它只记「这个目录应该指向上游仓库的 `386a11f` 这个 commit」。至于那个 commit 的实际内容在哪——在拼图 3。

> 类比 C++：`.gitmodules` 是**声明**（`extern`），gitlink 是**指针**（`T*`），拼图 3 是**实际数据**（堆上的对象）。

### 拼图 3：`.git/modules/` —— 实际数据

`.git/modules/` 是主仓库 `.git` 里的一个隐藏目录，每个子模块的**完整 `.git` 数据**存在这里：

```
.git/modules/_raw/rust-course/    ← 相当于一个完整的 git 仓库
    ├── HEAD
    ├── config
    ├── objects/                   ← commit、tree、blob 都在这里
    └── refs/
```

子目录 `_raw/rust-course/.git` 不是一个目录，而是一个**文本文件**，指向这里：

```
# cat _raw/rust-course/.git
gitdir: ../../.git/modules/_raw/rust-course
```

### 三块拼图的关系（一张图）

```
.gitmodules              gitlink (index)          .git/modules/_raw/xxx/
    ↓                        ↓                          ↓
 "url = github.com/      "这个 commit:             "386a11f 的完整
  sunface/rust-course"    386a11f"                 文件内容在这里"
```

> `.gitmodules` 告诉我**去哪找**，gitlink 告诉我**要哪个版本**，`.git/modules/` 存着**实际数据**。

## 日常操作（我真正用的到的）

### clone 别人的仓库（带子模块的）

```bash
# 方式 1：一步到位
git clone --recursive <url>

# 方式 2：先 clone 再补（适用于忘了 --recursive 的情况）
git clone <url>
cd <repo>
git submodule update --init --recursive
```

> `--init` 的意思是「读 `.gitmodules`，把 URL 写进 `.git/config`，然后 clone」。
> 如果不加 `--init`，git 只知道"有这个子模块"，但不知道去哪下载。

### 上游书更新了，我想同步

```bash
# 拉取每个子模块的最新 commit
git submodule update --remote

# 这会让子模块的 HEAD 指向上游最新，然后
# 主仓库的 gitlink（指针）也会变——需要 commit 一次：
git add _raw/rust-course
git commit -m "更新 rust-course 到最新"
```

### 新加一个子模块

```bash
git submodule add <url> <path>
# 这做了三件事：
# 1. clone 到 <path>（或复用已有的 clone）
# 2. 在 .gitmodules 加一条记录
# 3. 把 <path> 作为 gitlink 加进 index
```

### 删除一个子模块

```bash
git submodule deinit -f <path>    # 从 .git/config 移除
git rm <path>                      # 从 index 和 .gitmodules 移除
rm -rf .git/modules/<path>        # 清理实际数据
```

### 看子模块当前状态

```bash
git submodule status
# 输出示例：
#  386a11f17 _raw/rust-course (heads/main)
#  前面没有 - 或 + → 说明和工作区记录的 commit 一致
#  前面有 + → 说明子模块的 HEAD 和仓库记录的不一致
#  前面有 - → 说明子模块还没 init（空目录）
```

## 常见坑（我已经踩过或迟早会踩的）

### 坑 1：`git clone` 后子目录是空的

**原因**：忘了 `--recursive`。

**现象**：`_raw/rust-course/` 存在但是空目录。

**修**：`git submodule update --init --recursive`

### 坑 2：子模块里有未提交的改动

如果我在 `_raw/rust-course/` 里改了文件（比如加了批注），`git status` 会显示：

```
modified:   _raw/rust-course (new commits)
```

**注意**：子模块是只读引用上游的。如果我要改，应该先 fork 上游仓库，再把 submodule URL 改成我的 fork。别在 submodule 里直接改。

### 坑 3：切换分支后子模块不同步

从 `main` 切到 `feature` 分支后，`git submodule status` 可能显示 `+`：

```bash
git submodule update    # 让子模块回到当前分支记录的 commit
```

> 每次切换分支后跑一次 `git submodule update` 是最安全的习惯。

### 坑 4：`update --remote` 后忘记在主仓库 commit

`git submodule update --remote` 只更新了子模块的 HEAD，**主仓库的 gitlink 还指向旧的 commit**。如果我在主仓库 `git status`，会看到子模块有变更。**这意味着必须再 commit 一次主仓库**，否则别人 clone 的还是旧版本。

### 坑 5：代理 / 网络问题

子模块的 clone/fetch 走的是**子模块自己的 remote URL**，和主仓库一样需要网络。如果主仓库配了代理但子模块没继承——参考之前修过的 `http.proxy` 配置问题。

## 总结：和普通目录的本质区别

| | 普通目录 | submodule |
|---|---|---|
| git 跟踪什么 | 目录里的**每个文件** | 只记一个 **commit hash 指针** |
| 文件内容存在哪 | 主仓库的 `.git/objects/` | `.git/modules/<path>/objects/` |
| 别人 clone 后 | 自动有文件 | 空目录，需要 `--recursive` 或 `update --init` |
| 更新方式 | `git pull` | `git submodule update --remote` + 主仓库 commit |
| 适合什么 | 我自己的源码 | 引用别人的开源项目（只读） |
