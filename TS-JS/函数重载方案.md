# TypeScript 函数重载方案

> TS 不支持 C++ 式的多个独立函数体，但提供了声明合并 + 运行时单函数体的替代路径。本文从声明到实现，把四种分派方案讲透。
> 相关笔记：[[declare]]

---

## 1. TS 重载的本质：声明多，实现一

```ts
// 声明：编译期收窄调用签名
function add(a: number, b: number): number;
function add(a: string, b: string): string;

// 实现：运行时只有一个函数体
function add(a: number | string, b: number | string): number | string {
    if (typeof a === 'number' && typeof b === 'number') return a + b;
    if (typeof a === 'string' && typeof b === 'string') return a + b;
    throw new Error('type mismatch');
}
```

**关键认知**：TS 的重载是**编译期表态**——你告诉编译器"只能这么调我"，运行时只有一个函数体，型号信息在编译后全部消失。这和 C++ 的「编译后端根据参数选独立函数」完全不同。

---

## 2. 和 C++ / Rust 对照

| | C++ | TS | Rust |
|------|-----|------|------|
| 有没有函数重载 | ✅ 多个独立函数，编译时按参数类型分派 | ⚠️ 声明级——声明多个签名，一个实现体 | ❌ 没有，靠 trait 替代 |
| 有没有符号重载 | ✅ `operator+`，限制少 | ❌ 完全没有 | ✅ 通过 trait 实现，语义被锁住 |
| 运行时分派成本 | 零（编译时决定） | 零（但实现里要自己写 typeof 判断） | 零（trait 是编译期分派） |

## 3. 运行时实现——四种方案

### 方案 A：typeof 链（最简单，小规模 OK）

```ts
function handle(a: number | string, b: number | string) {
    if (typeof a === 'number' && typeof b === 'number') return a + b;
    if (typeof a === 'string' && typeof b === 'string') return a + b;
    throw new Error('type mismatch');
}
```

| 优点 | 缺点 |
|------|------|
| 零依赖，直接 | 3+ 个签名时 if 炸裂；新增重载要改核心函数 |

### 方案 B：字符串查表（参数数固定时还行）

```ts
const dispatch = {
    'number,number': (a: number, b: number) => a + b,
    'string,string': (a: string, b: string) => a + b,
} as const;

function handle(a: number | string, b: number | string) {
    const key = `${typeof a},${typeof b}`;
    const fn = dispatch[key as keyof typeof dispatch];
    if (!fn) throw Error(`no overload for (${key})`);
    return fn(a as never, b as never);
}
```

| 优点 | 缺点 |
|------|------|
| 分支清晰，新增重载只需加表项 | 每次调用拼字符串；高频调用有 GC 压力；`as any` 桥接丢类型 |

### 方案 C：嵌套对象查表（无字符串分配）

```ts
const dispatch = {
    number: { number: (a: number, b: number) => a + b },
    string: { string: (a: string, b: string) => a + b },
} as const;

function handle(a: number | string, b: number | string) {
    const fn = dispatch[typeof a]?.[typeof b];
    if (!fn) throw Error('type mismatch');
    return fn(a as never, b as never);
}
```

| 优点 | 缺点 |
|------|------|
| 零字符串分配，查两层 `Record` | 参数个数多时嵌套深 |

### 方案 D：canProcess 责任链（你项目里的方案 🔥）

```ts
interface ISubscriber {
    canProcess(...args: any[]): boolean;
    subscribe(info: SubscribeInstanceInfo, ...args: any[]): unknown;
}

class ActionSubscriber implements ISubscriber {
    canProcess(...args: any[]): boolean {
        return typeof args[0] === 'function' && isChildOf(args[0], Action);
    }
    subscribe(info, actionCtor, callback, thisArg) { ... }
}

class StoreActionSubscriber implements ISubscriber {
    canProcess(...args: any[]): boolean {
        return isChildOf(args[0], Store) && isChildOf(args[1], StoreAction);
    }
    subscribe(info, storeCtor, actionCtor, callback, thisArg) { ... }
}

// 调度器
class SubscribeHelper {
    private getSubscriber(...args: any[]) {
        for (let [_, v] of this.subscribers) {
            if (v.canProcess(...args)) return v;
        }
        throw Error('no matching subscriber');
    }
}
```

| 优点 | 缺点 |
|------|------|
| 零字符串、零查表；新增类型只需加类，不改调度器；`canProcess` 语义清晰 | 架构成本高；小项目过度设计 |

---

## 4. 方案选择指南

| 场景 | 推荐方案 | 为什么 |
|------|---------|--------|
| 2-3 个重载，偶尔调用 | A：typeof 链 | 最简 |
| 5-10 个，参数形态固定 | C：嵌套对象 | 零分配，查两层 |
| 参数个数/类型不确定 | 值得反思：是否真的需要这么多重载？ |
| 多种数据来源、每种有不同参数形态 | **D：canProcess 链** | 每种类型自己判断自己的参数是否合法 |
| C++ 桥接 Puerts | 每种参数形态独立 export | 不让「重载」穿透语言边界 |

---

## 5. 核心原则

- **重载签名是编译期防线，不要为了省写签名而塌缩成 `...args`**——一旦塌缩，调错参数编译期不报错，运行时才炸
- **运行时不要拼字符串做 key**——如果 `canProcess` 太重，至少用嵌套对象查表
- **如果实现体的分支已经复杂到需要拼动态键、处理不定长参数，你很可能在一个函数里塞了太多职责——该拆了**
