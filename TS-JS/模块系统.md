# TS/JS 模块系统

> JS 的模块加载是**运行时**的——这是理解一切模块问题的起点。C++ 的 `#include` 在编译期搞定，JS 的 `import` 在执行时一个一个加载文件。

## ESM vs CommonJS

JS 有两套模块系统。TS 源码用 `import/export` 语法（ESM），但编译后可以输出 CJS 或 ESM——取决于 `tsconfig.json` 的 `module` 设置。

| | ESM (`import`/`export`) | CommonJS (`require`/`module.exports`) |
|---|---|---|
| 年代 | ES6（2015） | Node.js 诞生起（2009） |
| 加载时机 | 编译期静态分析 → 运行时异步加载 | 运行时同步加载 |
| 导入的绑定 | **活绑定**（导出方变了，导入方自动看到） | **值拷贝**（导出方变了，导入方不变） |
| `this` 在顶层 | `undefined` | `module.exports` |
| 循环依赖 | 半成品对象，后续活着绑定 | 半成品对象，是快照 |
| Node 里能用吗 | `.mjs` 或 `"type":"module"` | `.cjs` 或默认 |
| C++ 类比 | 无直接对应 | 近似 `dlopen`/`LoadLibrary` |

### 活绑定 vs 值拷贝

```ts
// counter.ts
export let count = 0;
export function inc() { count++; }

// main.ts
import { count, inc } from './counter.js';
console.log(count);  // 0
inc();
console.log(count);  // 1 ← ESM 活绑定，自动看到变化

// === CJS 版本 ===
// counter.cjs
let count = 0;
function inc() { count++; }
module.exports = { count, inc };

// main.cjs
const { count, inc } = require('./counter.cjs');
console.log(count);  // 0
inc();
console.log(count);  // 0 ← CJS 是值拷贝！count 被导出时拷了一份
```

## 模块加载过程（ESM）

```
文件 main.ts:
  import { Hero } from './hero';
  import { Weapon } from './weapon';
                    ↓
         ┌─────────编译期──────────┐
         │ 1. 静态分析 import 语句  │
         │ 2. 构建模块依赖图        │
         │ 3. 找出所有入口         │
         └──────────┬─────────────┘
                    ↓
         ┌─────────运行期──────────┐
         │ 4. 深度优先加载模块     │
         │ 5. 每个模块只执行一次   │
         │ 6. 遇到循环 → 给半成品  │
         └────────────────────────┘
```

**第 6 步是关键**：引擎不报错，不重试——直接给一个"还没填充完"的 module 对象。后面依赖的变量等模块加载完后通过活绑定生效（ESM）或维持旧值（CJS）。

## 循环依赖

### 怎么炸的

```ts
// hero.ts
import { Weapon } from './weapon';
export class Hero {
  weapon!: Weapon;
}

// weapon.ts
import { Hero } from './hero';
export class Weapon {
  owner!: Hero;
}
```

加载时间线：

```
引擎加载 hero.ts
  → 发现 import Weapon → 暂停 hero，先加载 weapon.ts
    → weapon.ts 发现 import Hero → hero.ts 还没加载完！
    → 返回半成品 module 对象（此时 Hero 是 undefined）
  → hero.ts 恢复，Hero 被填回半成品 → Hero 活了
→ 但 Weapon 拿到的 Hero 引用通过活绑定也跟着活了
```

**前期不炸后期炸**的原因：加载顺序依赖文件被 `import` 的先后。删掉一个看似无关的 `import`，加载图变了，某个 `class` 被用到时恰好还没执行完——`undefined is not a constructor`。

### C++ 对照

```cpp
// C++: 前向声明解决循环
// hero.h
class Weapon;              // 前向声明 —— 不需要完整定义
class Hero {
  Weapon* weapon;          // 指针/引用不需要完整类型
};

// JS: 接口解决循环
// types.ts
interface IWeapon {}       // 接口 —— 没有运行时依赖
interface IHero { weapon: IWeapon; }
```

| | C++ | TS |
|---|---|---|
| 解决时机 | 编译期（预处理器） | 运行时 |
| 循环行为 | `#pragma once` 挡掉，最多需要前向声明 | 不报错，但可能拿到半成品 |
| 解法 | 前向声明 + 指针 | 接口 + 注册表 |

## 解决方案

### 方案 1：抽接口（最优）

```ts
// types.ts —— 零依赖
export interface IWeapon { fire(): void; }
export interface IHero { name: string; weapon: IWeapon; }

// hero.ts → 只 import types.ts
import { IWeapon } from './types';

// weapon.ts → 只 import types.ts
import { IHero } from './types';
```

两个具体类都依赖接口文件，互不 import。循环被接口切断。

### 方案 2：Registry（游戏里最实用）

```ts
// registry.ts —— 全局注册表，不 import 任何业务类
type Ctor = new (...args: any[]) => any;

class Registry {
  #m = new Map<string, Ctor>();
  reg(name: string, ctor: Ctor) { this.#m.set(name, ctor); }
  get(name: string): Ctor { return this.#m.get(name)!; }
}
export const registry = new Registry();

// hero.ts
import { registry } from './registry';
export class Hero {
  getWeapon() {
    const Weapon = registry.get('Weapon');  // 延迟获取
    return new Weapon(this);
  }
}
registry.reg('Hero', Hero);

// weapon.ts
import { registry } from './registry';
export class Weapon { constructor(public owner: any) {} }
registry.reg('Weapon', Weapon);
```

所有类只依赖 `registry`。需要对方时从注册表拿构造器。

### 方案 3：动态 import（临时止血）

```ts
// hero.ts
export class Hero {
  async getWeapon() {
    const { Weapon } = await import('./weapon');
    return new Weapon(this);
  }
}
```

运行时加载，不参与编译期依赖图。不适合高频调用。

## 命名空间式导出（barrel export）

C# 的 `namespace` 跨文件合并，TS 用文件夹 + barrel：

```
f/
├── a.ts   → export class A {}
├── b.ts   → export class B {}
└── index.ts  ← barrel
```

```ts
// f/index.ts
export { A } from './a';
export { B } from './b';

// 使用者
import * as F from './f';
F.A;  // class A
F.B;  // class B
```

> C# 编译期自动合并 namespace，TS 需要手动写 barrel 的 `export { X } from ...`。维护成本高一点，但效果一样。

## 和 C++ 的总对照

| | C++ | TS/JS |
|---|---|---|
| 模块机制 | `#include`（文本拼接，编译期） | `import`（运行时加载） |
| 跨文件组织 | namespace + 头文件 | barrel export + index.ts |
| 循环依赖解法 | 前向声明 + 指针 | 接口 + 注册表 |
| 导入的不变条件 | `#pragma once` 保证只解析一次 | 每个模块只执行一次，但时机不确定 |
| 改一行要重编译多少 | 改 `.h` → 所有 include 者重编译 | 只重编译改动的模块 |
