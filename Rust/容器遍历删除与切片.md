# 容器遍历删除与切片

## 边遍历边删除：为什么大家都头疼

根本原因不在语言，在数据结构。迭代器持有"我正在看这里"的游标，删除操作改变底层内存布局（元素迁移 / hash table rehash），游标原地的东西可能已经没了。

### 跨语言对比

| 语言 | 做法 | 代价 |
|---|---|---|
| **C++** `vector` | `it = vec.erase(it)` 返回下一个有效迭代器 | 能用，但每次 erase 是 O(n)，逐个删变成 O(n²)；正确做法是 erase-remove idiom |
| **C++** `map` | `it = map.erase(it)` 返回下一个 | O(log n) 每次，不需要搬元素，性能没问题 |
| **TS Map** | `for...of` 里 `map.delete(k)` 删当前 key | 删当前 key 安全；删还没遍历到的 key 会**静默跳过**它，不报错 |
| **Rust** | **编译期拒绝** | 迭代器持有不可变借用，删除需要可变借用 → 冲突，编译不过 |

Rust 的设计选择：**不给你"不拦你，但用错就 UB"的接口**。把安全路径封装成 `retain` / `drain`。

## `retain`：按条件过滤

```rust
// Vec: 保留奇数
let mut v = vec![1, 2, 3, 4, 5];
v.retain(|x| x % 2 != 0);
// → [1, 3, 5]

// HashMap: 保留 value > 10 的
map.retain(|_key, value| *value > 10);
```

- **闭包返回 `true` → 保留，`false` → 删除**（和 UE 的 `RemoveAll` 谓词方向相反）
- 内部对 Vec 做的是 erase-remove idiom——一趟 O(n)，不会 O(n²)
- 闭包里可以做副作用（记日志、收集被删元素的 id 等）：

```rust
let mut removed = Vec::new();
users.retain(|u| {
    if u.age < 18 {
        removed.push(u.id);
        false
    } else {
        true
    }
});
```

## `drain`：按范围抽取

把集合中某一段"抽走"——元素的所有权还给你，同时从原集合中删除。

```rust
let mut v = vec![1, 2, 3, 4, 5];

// drain 前半段
let drained: Vec<i32> = v.drain(0..2).collect();
println!("{:?}", drained); // [1, 2]
println!("{:?}", v);       // [3, 4, 5]

// 全部清空，同时拿到所有权
let all: Vec<i32> = v.drain(..).collect();
```

### `retain` vs `drain`

| | `retain` | `drain` |
|---|---|---|
| 删除的元素去哪了 | 直接 drop | 还给你，可以继续用 |
| 选择条件 | 闭包判断每个元素 | 按位置（range） |
| 典型场景 | 过滤掉不想要的 | 批量拿走一段去别处用 |

### UE 有对应吗？

没有直接等价。最接近的思路：对原数组 `RemoveAll` 的同时把要删的 push 到辅助队列——相当于一个闭包里同时做 retain 和 drain 的事。Rust 用 `drain` 一步到位。

## 切片 `&[T]`：连续内存的通用视图

### 什么是切片——胖指针

```rust
let arr = [1, 2, 3, 4, 5];
let slice: &[i32] = &arr[1..4]; // [2, 3, 4]

assert_eq!(slice.len(), 3); // 长度就在切片肚子里
```

```
切片 &[T]  =  { ptr, len }   ← 胖指针（fat pointer），两个字段打包在一起
裸指针 *const T  =  只有地址  ← 不知道能读多长
```

切片本身**不拥有数据**，只是借用一段连续内存的视图。

### Vec 和切片的关系：三层结构

```
Vec<i32>                    &[i32] (切片)
┌──────────┐                ┌──────────┐
│ ptr      │ ───────────→   │ ptr      │ ──→ [1][2][3][4][5]
│ len: 5   │   共同指向     │ len: 3   │     只看到其中一段
│ cap: 8   │                └──────────┘
└──────────┘                      ↑
   拥有所有权                     只是借用
```

- `Vec` 多了一个 `cap`（容量），知道还能塞多少元素不用重新分配
- `&[T]` 只有 `ptr + len`，不知道还剩多少空间

### Deref 自动转换

`Vec<T>` 实现了 `Deref<Target = [T]>`，所以接受 `&[T]` 的地方，`&Vec<T>` 会自动变过去：

```rust
fn good(slice: &[i32]) { } // ✅ 通用

let v = vec![1, 2, 3];
let a = [1, 2, 3];
let s = &v[1..];

good(&v);  // ✅ Vec → 切片
good(&a);  // ✅ 数组 → 切片
good(s);   // ✅ 切片 → 切片
```

**函数参数优先用 `&[T]` 而不是 `&Vec<T>`**——它可以接受 Vec、数组、其他切片的引用。

## 三层：数组 / Vec / Box<[T]>

```rust
[T; N]         // 编译期定长，栈上（或内联在结构体里）
Vec<T>         // 可增长，堆上，有 cap
Box<[T]>       // 运行时定长但不可增长，堆上，没有 cap
```

```
Vec<T>                    Box<[T]>
┌──────────┐              ┌──────────┐
│ ptr      │              │ ptr      │
│ len: 5   │              │ len: 5   │
│ cap: 8   │              └──────────┘ ← 没有 cap！
└──────────┘                    
```

### `Box<[T]>` 什么时候用

当数组确定永远不会再增长，可以用 `into_boxed_slice` 把多余的容量释放掉，省一个 `usize` 的空间：

```rust
let v = vec![1, 2, 3, 4, 5];
let b: Box<[i32]> = v.into_boxed_slice(); // cap 收缩到恰好 len
// b.push(6); // ❌ Box<[T]> 没有 push
```

### `Box<[i32; 5]>`：编译期定长在堆上

```rust
let b: Box<[i32; 5]> = Box::new([1, 2, 3, 4, 5]);
//     ^^^^^^^^^^^^^ 长度写在类型里
```

语义上先栈上构造数组，再拷到堆上（release build 下 LLVM 可能优化掉中间拷贝）。`vec![]` 宏则直接在堆上分配，不经过栈——如果关心"堆上直接构造"，用 `vec!` 比 `Box::new([...])` 更直接。

### 跨语言对照

| Rust | C++ |
|---|---|
| `[T; N]` | `std::array<T, N>` / `T[N]` |
| `Vec<T>` | `std::vector<T>` |
| `Box<[T]>` | `std::unique_ptr<T[]>`（接近，但不带长度） |
| `&[T]` | `std::span<T>`（C++20） |

> **注意**：`Box<[T]>` ≠ `std::array`。`std::array` 的大小是编译期模板参数，元素在栈上，对应的是 Rust 的 `[T; N]`。不要搞混。

## Deref 和 auto-deref

### 基本行为

`Box<T>` 实现了 `Deref<Target = T>`，所以可以直接用 `.` 访问内部字段和方法：

```rust
let b = Box::new(Point { x: 3, y: 4 });
println!("{}", b.x);   // ✅ 直接 . 访问，不需要 get 或 *
println!("{}", b.len()); // ✅ 直接调方法
```

编译器在遇到 `.` 的时候，如果当前类型没找到对应字段/方法，会顺着 `Deref` 链往下找。**这不是运算符重载**——Rust 不能重载 `.`，`Deref` 只是一个 trait，告诉编译器"往这看"。

### 和 C++ 的区别

| | 字段访问 | 方法调用 |
|---|---|---|
| C++ `unique_ptr<T>` | `p->x` | `p->method()` |
| Rust `Box<T>` | `b.x` | `b.method()` |

Rust 没有 `->` 操作符，`.` 配合 auto-deref 把两者的活全干了。

### `&Box<T>` → `&T` 的小坑

`&Box<T>` 在目标类型匹配 `&T` 时，会自动 deref 变成 `&T`：

```rust
let b: Box<i32> = Box::new(42);
let r: &i32 = &b; // &Box<i32> → &i32，自动
```

这会导致三个反直觉的点：

1. **比较走值比较，不是指针身份**：
   ```rust
   let b1 = Box::new(42);
   let b2 = Box::new(42);
   assert!(b1 == b2); // true —— 比的是 42 == 42，不是指针地址
   ```
   和 C++ 不一样，C++ 那边 `p1 == p2` 比指针，要比值得写 `*p1 == *p2`。

2. **方法调用歧义**：如果 `Box` 后期加了和你类型同名的方法，代码会静默改用 `Box` 的方法。不过标准库刻意避免了这种冲突。

3. **拿不到 Box 本身在栈上的地址**：
   ```rust
   fn wants_box_ref(x: &Box<i32>) { }
   wants_box_ref(&b); // 这个可以 —— 类型签名明确要求 &Box<i32> 时不 deref
   ```
   编译器"看目标类型再决定要不要 deref"，不是盲目 deref。把类型写清楚就能避开。
