# 结构体更新、移动语义与智能指针

> 上次笔记到 `()` unit 类型为止。这次聊了：`..` 结构体更新语法的所有权细节、Rust 移动 = memcpy + 编译器标记、为什么不需要右值引用、智能指针全景、内部可变性（`Cell`/`RefCell`）、生命周期标注的日常策略、`Rc` 循环泄漏与 `Weak`。

## `..` 结构体更新：不是 TS spread，是字段级移动展开

```rust
let user2 = User {
    email: String::from("another@example.com"),
    ..user1
};
```

`..user1` 是**语法糖**——编译器把它展开为逐字段的 `field: user1.field`。先处理显式字段、再 `..` 补位，重复字段编译报错。

**属性是 `Copy` 的字段**（`bool`、`u64`）：复制走，原字段还能用。  
**非 `Copy` 字段**（`String`）：所有权移走，原字段报废。  
**整个 struct**：只要有任意字段被移走，整体标记为 `PARTIALLY_MOVED`，不能再整体使用。但单独访问"剩下"的字段仍然 OK。

```rust
println!("{}", user1.active);   // ✅ Copy 了，还在
println!("{}", user1.username); // ❌ 所有权被移走了
let user3 = user1;              // ❌ 整体已部分移动
```

## Rust 移动 = memcpy + 编译器标记（没有移动构造）

```rust
let a = String::from("hello");
let b = a;  // 就是 24 字节（ptr+len+cap）拷贝到 b，标记 a 不可用
```

**堆上数据从不动。** 栈上的 String 三个字可以搬无数次，堆上 `"hello"` 只分配一次。没函数调用、没回调、没析构多余执行。

### C++ 为什么需要移动构造、Rust 不需要

C++：`std::move(a)` 后 `a` 还活着，析构照跑。如果不把 `a` 内部指针置空 → double free。必须写移动构造**手动清理源对象**。

Rust：移动后 `a` **就没了**（编译器不让你碰）。没有析构要跑，不需要置空。**移动永远是 memcpy，省了清空源对象那一步。**

| | C++ | Rust |
|---|---|---|
| 移动机制 | 调用移动构造**函数** | memcpy |
| 源对象状态 | "valid but unspecified"（自己保证） | 编译器标记不可用 |
| 能否自定义 | ✅ | ❌ |
| 右值引用 | 必须（触发移动构造） | 不需要（源对象直接报废） |

---

## 为什么 Rust 不会引入右值引用

C++ 右值引用存在的理由：**移动后源对象还要析构，必须有人清理它。** Rust 把这个前提删了——源对象直接不可用。所以不需要右值引用，也不会引入。

极致性能的瓶颈不在「能不能自定义移动」——大对象用 `Box` 放堆上、用借用、LLVM 自动 elision，都比右值引用优雅。

## 智能指针全景

```
无所有权（借用）
├── &T           不可变借用
└── &mut T       可变借用

有所有权
├── Box<T>        堆分配，独占               ≈ unique_ptr<T>
├── Rc<T>         引用计数，单线程            ≈ shared_ptr<T>（非原子）
├── Arc<T>        引用计数，多线程（原子）     ≈ shared_ptr<T>（原子版）
└── Weak<T>       不增计数的弱引用            ≈ weak_ptr<T>

内部可变性（&self 下修改值）
├── Cell<T>        整体替换（Copy），零开销
├── RefCell<T>     运行时借用检查，单线程
└── Mutex<T>       多线程版 RefCell
```

---

## `Cell<T>`：「永远不暴露内部引用」换「&self 改值」

```rust
let x = Cell::new(42);
x.set(99);          // 拿了 &self，改了值
let val = x.get();  // Copy 一份出来
```

关键：**你永远拿不到 `&T` 或 `&mut T` 指向内部数据。** `set` 内部用 `mem::replace` 一瞬间见到 `&mut` 塞完立刻收手——别名问题不存在。所以不破坏借用规则——只是换了一种保证方式。

---

## `RefCell<T>`：编译期检查推到运行时

```rust
let x = RefCell::new(vec![1, 2, 3]);
let r = x.borrow();            // 计数 +1
let m = x.borrow_mut();        // 💥 panic! 已有不可变借用
```

成本：`borrow()` = **1 次读 + 1 次 if + 1 次写**，约 2-4 ns。非原子指令，无锁、无 syscall。比 `Mutex` 轻一个数量级。

| | 编译期借用 | `RefCell` | `Mutex` |
|---|---|---|---|
| 成本 | 0 | ~2-4 ns | ~20-50 ns（无竞争） |
| 失败结果 | 编译错误 | panic | 阻塞等待 |
| 线程安全 | 单/多均可 | 单线程（`!Sync`） | 多线程 |

## 生命周期标注：什么时候该写、什么时候可以躲

| 场景 | 做法 |
|---|---|
| 结构体字段 = 引用（`&'a str`） | **必须标生命周期** |
| 两个字段间有引用关系 | **必须标** |
| 迭代器包装另一个迭代器 | **必须标** |
| 日常字段：`String`、`Vec`、`i32` | owned 类型直接持有，不标 |
| 需要共享所有权但不想标生命周期 | `Rc<T>` 套娃 |

日常策略：**能 owned 不 borrow，能 `Rc` 不 `&`。** 生命周期标注集中在库代码和底层抽象，应用层很少碰到。

## `Rc` 循环泄漏：编译器看不见

```rust
*a.next = Some(b.clone());  // a → b，b 计数 +1
*b.next = Some(a.clone());  // b → a，a 计数 +1
// 离开作用域：各剩 1，都不释放 → 泄漏
```

rustc 全程沉默。**`Rc` 循环是安全的内存泄漏——不是 UB。** 编译器只保证 UAF 和 double free 不发生，不保证绝对不泄漏。

解法：用 `Weak<T>` 断开其中一个方向：

```rust
struct Node {
    parent: Weak<Node>,        // 弱引用，不增加计数
    children: Vec<Rc<Node>>,   // 强引用，单向
}
```

树状所有权（强引用单向）→ 不构成环。

## JSON 解析与 serde

JSON 是树不是图 → 不需要 `Rc`。Rust 标准答案：

```rust
#[derive(Deserialize, Debug)]
struct User {
    name: String,
    age: u8,
    tags: Vec<String>,
}

let user: User = serde_json::from_str(json_str).unwrap();
// 自动按字段名绑定，等价 TS 的 interface 接 JSON
```

核心数据结构是递归 enum（`Value::Array(Vec<Value>)`），所有权单向向下，天然无环。可自定义重命名、默认值、缺省处理（`#[serde(rename)]`、`#[serde(default)]` 等）。
